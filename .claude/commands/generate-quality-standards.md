# Generate Quality Standards

Analyze the current project's codebase to generate a project-specific `.claude/QUALITY.md` file with discovered patterns and conventions.

## Why Project-Specific

- Different projects use different tech stacks
- Coding conventions vary by team
- A generic global file may contradict actual practices
- Standards should reflect what the team actually does

## Process

### Step 1: Discover Tech Stack

Analyze manifest files to determine technologies:

```
package.json     → Node.js, React, testing framework
pom.xml          → Java, Spring Boot
go.mod           → Go modules
Cargo.toml       → Rust
pyproject.toml   → Python
```

### Step 2: Extract Code Style

Read configuration files:

| File            | Extracts                  |
| --------------- | ------------------------- |
| `.eslintrc*`    | Linting rules, code style |
| `.prettierrc*`  | Formatting preferences    |
| `tsconfig.json` | TypeScript strictness     |
| `.editorconfig` | Indentation, line endings |
| `biome.json`    | Modern linting rules      |

### Step 3: Analyze Testing Patterns

Search test files for conventions:

- Test file naming (`*.spec.ts`, `*.test.ts`, `*_test.go`)
- Test structure (describe/it, test suites)
- Mock patterns (what's being mocked, how)
- Coverage requirements (from CI config)

### Step 4: Extract Naming Conventions

Sample actual code to find patterns:

- Variable naming (camelCase vs snake_case)
- Function naming (verb prefixes, etc.)
- File naming (kebab-case, PascalCase)
- Component naming (React/Angular patterns)

### Step 5: Identify Common Patterns

Look for established patterns:

- Error handling (try/catch, Result types, error boundaries)
- Logging (what logger, format)
- State management (Redux, Zustand, Context)
- API patterns (REST, GraphQL, tRPC)

### Step 6: Generate QUALITY.md

Create `.claude/QUALITY.md` in project root with:

```markdown
# Code Quality Standards - [Project Name]

> Auto-generated by `/generate-quality-standards` on [Date]
> Based on analysis of actual codebase patterns

## Tech Stack

[Discovered from manifest files]

## Code Style

[Extracted from linter/formatter configs]

## Naming Conventions

[Observed from code samples]

### Examples from Codebase

[Real examples from this project]

## Testing Requirements

[From test file analysis]

### Test Patterns Used

[Actual patterns found]

## Common Patterns

[Discovered patterns with file references]

## Anti-Patterns to Avoid

[Based on linter rules and observed practices]
```

## Output

Creates or updates: `.claude/QUALITY.md`

If file exists, shows diff and asks for confirmation before overwriting.

## When to Run

- Starting work on a new project
- After major refactors
- When onboarding to unfamiliar codebase
- When `@code-reviewer` needs updated standards

## Integration

- `@documentation-updater` can invoke this when detecting pattern changes
- `@code-reviewer` references the generated `.claude/QUALITY.md`
- `/project-reflection` may suggest running this if conventions are unclear
